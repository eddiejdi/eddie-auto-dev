# Operações 2026-05-16 — Fita LTO, Drain e Ajustes de Alerta

## Resumo

Quatro ações operacionais realizadas em 2026-05-16 cobrindo a fita LTO NC2508, o pipeline de drain, alertas Prometheus e o Nextcloud.

## 1. Formatação da Fita NC2508

### Contexto

A fita NC2508 passou por `ltfsck --full-recovery` em 2026-05-14 e múltiplas tentativas de drain em 2026-05-15. O drain foi interrompido por timeout do systemd antes de commitar os dados ao índice LTFS. Ao remontar a fita em 2026-05-16, o `df` mostrou apenas 46M (dados de sessões anteriores em buffer RAM não persistidos) em vez dos 27G esperados.

### Decisão

Formatar a fita NC2508 do zero para eliminar estado inconsistente e retransferir os snapshots completamente.

**Dados perdidos:** parciais não commitados (46M físicos + buffer RAM). Os snapshots originais (`rpa4all-snapshot-20260501T030006Z` 59G e `rpa4all-snapshot-20260502T100642Z` 99G) estão intactos em `/mnt/raid1/backups/` no homelab.

### Execução

```bash
# Na NAS (192.168.15.4), via homelab:
mkltfs --device=/dev/sg0 --tape-serial=NC2508 --force
```

**Resultado:**
- UUID: `31cb13b4-4a7c-4f72-87b4-43f86ead48e3`
- Gen: 1
- Capacidade disponível: 2524 GB

## 2. Drain Reiniciado com TimeoutStartSec=infinity

### Problema

O `lto6-drain-backups.service` tinha `TimeoutStartSec=7200` (2h). O rsync de 59G via LTFS/CIFS levou mais de 2h → systemd matou o processo com SIGKILL → LTFS não teve tempo de commitar o buffer.

### Fix

```ini
# /etc/systemd/system/lto6-drain-backups.service
TimeoutStartSec=infinity
```

**Por quê `infinity`:** LTFS sobre fita LTO é inerentemente lento e não tem timeout previsível para volumes grandes. O drain deve completar sem restrição de tempo do systemd.

### Estado

Drain ativo desde 19:34 transferindo `rpa4all-snapshot-20260501T030006Z` → fita via CIFS. Ao concluir, avança automaticamente para `rpa4all-snapshot-20260502T100642Z`.

## 3. Alerta NASLTFSDependencyDown — Tolerância 2m → 5m

### Problema

O alerta `NASLTFSDependencyDown` disparou durante o `systemctl stop ltfs-lto6.service`, quando o serviço demorou brevemente para parar (rsync em estado D travou o CIFS). A NAS ficou inacessível por menos de 2 minutos → falso positivo.

### Fix

Arquivo: `monitoring/prometheus/ltfs-selfheal-rules.yml`

```yaml
# Antes
for: 2m

# Depois
for: 5m
```

A descrição da anotação também foi atualizada para refletir "5 minutos".

**Prometheus recarregado via:** `docker kill -s HUP prometheus`

## 4. Nextcloud — Bruteforce Protection Allowlist

### Problema

O app Android (auto-upload de screenshots) disparou muitas requisições DAV em rápida sucessão → Nextcloud bloqueou o IP `192.168.15.150` por bruteforce protection → HTTP 429 em todas as chamadas.

### Fix

```bash
# Liberar o IP bloqueado imediatamente:
docker exec nextcloud-app php occ security:bruteforce:reset 192.168.15.150

# Evitar bloqueios futuros de dispositivos internos:
docker exec nextcloud-app php occ config:system:set \
  bruteForceProtection.allowlist 0 --value='192.168.15.0/24'
```

**Resultado:** rede local `192.168.15.0/24` isenta de bruteforce protection. Proteção continua ativa para IPs externos.

## Lições Aprendidas

1. **LTFS buffer ≠ dados na fita.** O `df` do LTFS mostra o espaço do índice físico. Dados em buffer RAM são perdidos se o unmount não for gracioso. Sempre aguardar o `ltfs-cache-flush` antes de parar o serviço.

2. **`TimeoutStartSec` em serviços de I/O de fita** deve ser `infinity`. Qualquer valor fixo é arbitrário e pode causar perda de dados por SIGKILL.

3. **Bruteforce protection do Nextcloud** bloqueia IPs internos se o app entrar em loop de retry. Adicionar a subnet local à allowlist é seguro e evita falsos bloqueios.
